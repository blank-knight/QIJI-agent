# 离线包安装速度优化方案

**文档版本**: v2.0（已实施）
**创建日期**: 2026-07-07
**最后更新**: 2026-07-07
**适用范围**: 奇计Claw Windows 离线安装包
**实施状态**: ✅ 方案 A + B + C 已实施，代码 dirty 待测试

---

## 问题背景

用户反馈奇计Claw 离线包（NSIS 打包，713MB）在 Windows 上安装非常慢，耗时 **5-10 分钟**。

### 问题描述

1. **安装时间过长**：首次安装或更新时耗时 5-10 分钟
2. **用户体验差**：长时间等待无进度反馈
3. **更新功能依赖慢**：桌面端的"检查更新"功能也受影响

---

## 根本原因分析

通过源码分析，慢的原因有三个：

### 1. git add -A 索引 99,184 文件

**位置**: `install.ps1` Stage-VendorFiles 函数内

```powershell
git -c windows.appendAtomically=false add -A 2>$null
git -c windows.appendAtomically=false commit -m "vendor snapshot" 2>$null
```

**问题**：
- hermes-agent 仓库有 **99,184 个文件**
- `git add -A` 需要扫描整个目录并为每个文件建立索引
- 在 Windows 文件系统（NTFS）上，这个操作耗时 **2-3 分钟**

**耗时**: 2-3 分钟

### 2. Windows Defender 实时扫描

**机制**：
- Windows Defender 监控所有文件写入操作
- 每个文件被写入/复制时都会被扫描
- Defender 对"未知来源"的脚本不信任，扫描频率高

**影响**：
- robocopy 复制大量文件（git、node、python、site-packages）
- 每个文件写入都会触发 Defender 扫描
- Defender 开启时，安装时间 **慢 5-10 倍**

**耗时**: 2-5 分钟（取决于 Defender 配置）

### 3. robocopy 单线程复制 + 重试过于激进

**位置**: `install.ps1` Stage-VendorFiles 函数

```powershell
& robocopy $vendorSitePackages "$managedVenv\Lib\site-packages" /E /NJH /NJS /NFL /NDL /NP /R:5 /W:3 2>$null
```

**问题**：
- **单线程复制**：site-packages 5万+文件、node_modules 5万+文件全部串行
- **重试太激进**：每个文件锁住时重试 5 次 × 等 3 秒 = 15 秒，离线安装文件不会被锁
- 8 处 robocopy 全部是这个配置

**耗时**: 1-2 分钟

## 时间分解

### 优化前

| 环节 | 耗时 | Defender 影响 | 可优化 |
|------|------|---------------|--------|
| robocopy 大文件复制（8处） | 1-2 分钟 | ⚠️ 扫描每个文件 | ✅ 多线程 |
| git add -A（99,184 文件） | 2-3 分钟 | ⚠️ 索引时扫描 | ✅ 只 add 关键文件 |
| git commit | 5 秒 | ✅ 快 | ✅ 只 commit 关键文件 |
| uv pip install 修复 | ~2 秒 | ✅ 快 | ❌ 无需优化 |
| **总计（Defender 开启）** | **5-10 分钟** | - | - |
| **总计（Defender 关闭）** | **3-5 分钟** | - | - |

### 优化后（预期）

| 环节 | 耗时 | 说明 |
|------|------|------|
| robocopy（8处，多线程+减少重试） | 30-60 秒 | /MT 动态线程 + /R:1 /W:1 |
| git add（4 个关键文件） | 5 秒 | 只 add pyproject.toml 等 |
| git commit | 5 秒 | 不变 |
| uv pip install 修复 | ~2 秒 | 不变 |
| **总计（Defender 已排除）** | **1-2 分钟** | /MT:16 全速并行 |
| **总计（Defender 未排除）** | **3-4 分钟** | /MT:CPU/2 保守并行 |

## 已实施方案

### 方案 A：优化 git add（✅ 已实施）

**思路**：只添加关键文件，不索引全部 99,184 文件

**实际修改**（`install.ps1` 第 205 行）：
```powershell
# 修改前
git -c windows.appendAtomically=false add -A 2>$null
git -c windows.appendAtomically=false commit -m "vendor snapshot" 2>$null

# 修改后
git -c windows.appendAtomically=false add pyproject.toml hermes_cli/__init__.py AGENTS.md README.md 2>$null
git -c windows.appendAtomically=false commit -m "vendor snapshot (offline install)" 2>$null
```

**选择的 4 个文件**：`pyproject.toml`、`hermes_cli/__init__.py`、`AGENTS.md`、`README.md`
（选这 4 个是因为它们在源码中必定存在；`.install_method` 是安装时才生成的，源码里没有）

**为什么不影响在线更新**：
- `git rev-parse --verify HEAD` 只检查是否有初始提交（不关心提交内容）
- `git checkout main` 基于远程分支（不关心本地提交内容）
- `git pull --ff-only origin main` 基于远程分支（不关心本地提交内容）
- 用户运行 `hermes update` 时，git fetch + checkout 会覆盖工作树为远程内容

**效果**：2-3 分钟 → **5-10 秒**

### 方案 B：Defender 排除 + 动态线程检测（✅ 已实施）

**思路**：在 Stage-VendorFiles 之前（不是之后），检测 Defender 状态，动态决定线程数

**关键设计**：Defender 排除必须在 robocopy 之前生效，否则排除加了也赶不上文件复制

**实际代码**（`install.ps1` 第 370-430 行，Stage-VendorFiles 调用之前）：

```powershell
$script:RobocopyMT = 8  # sensible default

if ($env:OS -eq "Windows_NT") {
    # 检测 Defender 状态
    $status = Get-MpComputerStatus -ErrorAction SilentlyContinue
    $prefs  = Get-MpPreference -ErrorAction SilentlyContinue

    $defenderScanning = (
        $status -and
        $status.RealTimeProtectionEnabled -and
        -not ($prefs -and ($prefs.ExclusionPath -contains $HermesHome))
    )

    if ($defenderScanning) {
        # Defender 在扫描 $HermesHome
        if ($isAdmin) {
            # 管理员 → 自动排除 + 16线程
            Add-MpPreference -ExclusionPath $HermesHome
            $script:RobocopyMT = 16
        } else {
            # 非管理员 → 提示手动加 + CPU核心数/2线程
            $script:RobocopyMT = [Math]::Max(2, [int]($env:NUMBER_OF_PROCESSORS / 2))
        }
    } else {
        # Defender 关闭或已排除 → 16线程全速
        $script:RobocopyMT = 16
    }
}
```

**动态线程数逻辑**：

| 场景 | 线程数 | 原因 |
|------|--------|------|
| Defender 已排除 / 已关闭 | 16 | 没有扫描开销，全速并行 |
| Defender 未排除 + 管理员 | 16 | 自动排除后没有扫描开销 |
| Defender 未排除 + 非管理员 | CPU核心/2 | 保守并行，避免扫描争抢 CPU |
| 非 Windows 系统 | 8（默认） | 不影响（robocopy 只在 Windows） |

**为什么非管理员时用 CPU/2**：
- 4 核机器：2 线程（Defender 每个并发扫描占一个核心）
- 8 核机器：4 线程
- 留一半 CPU 给系统 + Defender 扫描

**效果**：
- 管理员运行：安装时间 **1-2 分钟**
- 普通用户加了排除：**2-3 分钟**
- 普通用户没加排除：**3-4 分钟**

### 方案 C：robocopy 多线程 + 减少重试（✅ 已实施）

**思路**：所有 8 处 robocopy 调用统一优化

**实际修改**（`install.ps1` 8 处 robocopy，批量替换）：

```powershell
# 修改前（8 处全部如此）
& robocopy $src $dst /E /NJH /NJS /NFL /NDL /NP /R:5 /W:3 2>$null | Out-Null

# 修改后
& robocopy $src $dst /E /NJH /NJS /NFL /NDL /NP /R:1 /W:1 /MT:$script:RobocopyMT 2>$null | Out-Null
```

**两处改动**：
1. `/R:5 /W:3` → `/R:1 /W:1`：重试从 15 秒/文件降到 1 秒/文件（离线安装文件不会被锁）
2. 新增 `/MT:$script:RobocopyMT`：多线程并行复制（线程数由方案 B 动态决定）

**涉及的 8 处 robocopy**：

| 序号 | 源目录 | 文件量 | 说明 |
|------|--------|--------|------|
| 1 | vendor\git | ~3,000 | PortableGit |
| 2 | vendor\node | ~5,000 | Node.js runtime |
| 3 | vendor\hermes-agent | ~99,000 | 源码（但只复制，不 git add） |
| 4 | vendor\python | ~1,000 | Python 解释器 |
| 5 | vendor\site-packages | ~50,000+ | pip 依赖（最大） |
| 6 | vendor\venv-scripts | ~50 | 入口点 |
| 7 | vendor\nm | ~50,000+ | node_modules（最大） |
| 8 | vendor\chromium | ~1,000 | Playwright 浏览器 |

**效果**：1-2 分钟 → **30-60 秒**（多线程 + 减少重试）

### Defender 排除为什么有效？

**Defender 的工作机制**：
- 监控所有文件 I/O 操作（读取、写入、执行）
- 对每个操作进行扫描（签名检查、启发式分析）
- 对"未知来源"的操作扫描频率高

**添加排除后的行为**：
- Defender 不扫描 `$HermesHome` 下的所有文件
- robocopy、git add 等操作不会被扫描
- 安装速度提升 **5-10 倍**

**安全性**：
- `$HermesHome` 是奇计Claw 的安装目录
- 只包含奇计Claw 自己的文件（python、git、hermes-agent 源码）
- 不会有其他程序写入这个目录
- 安全风险低

---

### robocopy /MT 为什么需要动态？

**robocopy /MT 的开销**：
- 每个线程很轻量（几 MB 内存），CPU 占用不高
- 但 Defender 的实时扫描会随线程数线性增加：16 线程 = 最多 16 个并发扫描

**4 核 8GB 机器的情况**：

| /MT 值 | Defender 状态 | 实际效果 |
|--------|-------------|---------|
| 16 | 已排除 | ✅ 完美，全速并行 |
| 16 | 未排除 | ❌ 16 个并发扫描争抢 4 核 CPU，可能更慢 |
| 4 | 未排除 | ⚠️ 偏高，4 个扫描占满 CPU |
| 2 | 未排除 | ✅ 合适，留 2 核给系统 |

**动态检测的逻辑**：
- Defender 已排除 → 16 线程（没有扫描开销）
- Defender 未排除 + 管理员 → 自动排除后 16 线程
- Defender 未排除 + 非管理员 → CPU核心/2 线程（保守）

**新增代码**（在 Stage-VendorFiles 调用之前）：
```powershell
$script:RobocopyMT = 8  # sensible default

if ($env:OS -eq "Windows_NT") {
    try {
        $status = Get-MpComputerStatus -ErrorAction SilentlyContinue
        $prefs  = Get-MpPreference -ErrorAction SilentlyContinue
    } catch { $status = $null; $prefs = $null }

    $defenderScanning = (
        $status -and
        $status.RealTimeProtectionEnabled -and
        -not ($prefs -and ($prefs.ExclusionPath -contains $HermesHome))
    )

    if ($defenderScanning) {
        $isAdmin = ([Security.Principal.WindowsPrincipal]::new(
            [Security.Principal.WindowsIdentity]::GetCurrent()
        )).IsInRole([Security.Principal.WindowsBuiltInRole]::Administrator)

        if ($isAdmin) {
            try {
                Add-MpPreference -ExclusionPath $HermesHome -ErrorAction Stop
                Write-Host "[Defender] Added exclusion for $HermesHome (elevated)" -ForegroundColor Green
                $script:RobocopyMT = 16
            } catch {
                Write-Host "Tip: Windows Defender real-time scanning can slow this install significantly." -ForegroundColor Yellow
                Write-Host "     To speed up (5-10x faster), add an exclusion in an admin PowerShell:" -ForegroundColor Yellow
                Write-Host "     Add-MpPreference -ExclusionPath `"$HermesHome`"" -ForegroundColor Cyan
                $script:RobocopyMT = [Math]::Max(2, [int]($env:NUMBER_OF_PROCESSORS / 2))
            }
        } else {
            Write-Host "Tip: Windows Defender real-time scanning can slow this install significantly." -ForegroundColor Yellow
            Write-Host "     To speed up (5-10x faster), add an exclusion in an admin PowerShell:" -ForegroundColor Yellow
            Write-Host "     Add-MpPreference -ExclusionPath `"$HermesHome`"" -ForegroundColor Cyan
            $script:RobocopyMT = [Math]::Max(2, [int]($env:NUMBER_OF_PROCESSORS / 2))
        }
    } else {
        $script:RobocopyMT = 16
    }
}

Write-Host "[robocopy] Using $script:RobocopyMT threads" -ForegroundColor DarkGray
```

## 注意事项

### 1. 离线包更新机制的影响

**问题**：如果未来去掉 git（方案 D），桌面端更新会失败

**当前状态**：方案 A 只优化了 git add 的文件数，保留了 git 仓库结构，不影响更新

### 2. Windows Defender 企业策略

**问题**：企业环境可能禁止修改 Defender 策略

**当前处理**：
- 自动排除失败时不报错，降级为保守线程数
- 提示信息仍然显示，用户可联系 IT 管理员手动添加

### 3. robocopy /MT 兼容性

**问题**：极旧版本的 Windows（XP/Server 2003）的 robocopy 不支持 /MT

**当前状态**：
- 奇计Claw 要求 Windows 10+，robocopy 版本支持 /MT
- 如果 /MT 参数不被识别，robocopy 会报错退出（不会静默失败）

### 4. 非 Windows 系统

**问题**：install.ps1 只在 Windows 运行，但脚本可能被误执行

**当前处理**：
- Defender 检测块用 `if ($env:OS -eq "Windows_NT")` 保护
- robocopy 在非 Windows 系统上不存在，会直接报错（预期行为）

## 未来优化方向

### 1. site-packages / node_modules 预压缩（可选）

**思路**：vendor 目录里存 7z 压缩包，安装时解压

**优势**：
- Defender 只扫描 1 个 7z 文件（而不是 5 万个小文件）
- vendor 目录体积减小

**劣势**：
- 需要修改 prepare-offline.ps1 打包逻辑
- 解压仍然触发 Defender 扫描
- 需要测试 7z 解压 + Defender vs robocopy + Defender 哪个快

### 2. 增量更新

**思路**：只下载变更的文件（配合方案 D）

**优势**：更新时下载量小

**劣势**：需要维护变更历史，实现复杂

### 3. 签名安装器

**思路**：购买代码签名证书，签名 NSIS 安装器

**优势**：Defender 扫描频率进一步降低，用户体验更好

**劣势**：需要预算（$200-500/年）
